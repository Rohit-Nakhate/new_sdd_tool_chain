Internal Component Design

Internal Component Design

The component has the following files that are developed by Continental:

  • DWA_Task.c - This file will init, run and stop the DWA functionality
  • DWA_Task.h - Header file for DWA_Task.c
  • DWA_NvRam.h - Header file is needed for connecting memory and lib
  • DWA_ExtraDefinitions.h - This headerfile controls the overwriting of generated S/R autosar_functions
  • DWA_DataTypes.h - This headerfile provides the datatypes used to provide the S/R Autosar functions for coding, fault memory, etc.
  • DWA_MemMap.h - Header file used during RTE generation

Interoperability Interaction Criticality Technical complexity Risks Testability

Interoperability

Interoperability is set to low because DWA depends on other components.

Interaction

Interaction is set to low because the component has a reduced number of non RTE dependencies.

Criticality

Since DWA is non-safety, the interaction is Low and the component was not integrated recently, the criticality is medium.

Technical complexity

The technical complexity is medium because DWA implements a specific behavior.

Risks

The component DWA is composed of only one unit and no complex alghoritms are implemented therefore the risk is low.

Testability

The component DWA requires other tools (like PLAST) besides SWATT for 100% tests coverage but all tests can be automated. So, the testability level is high.
