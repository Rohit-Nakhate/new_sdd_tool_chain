Internal Component Design

Internal Component Design

The component is splitted into the follwing SW units:

  • ZVAD_Task.c - Code is called by scheduler which will init, start and stop the operation of ZVAD
  • ZVAD_Task.h - Header file for ZVAD_Task.c
  • ZVAD_NvRam.h - Header file is needed for connecting memory and lib
Internal Component Design

Requirements Allocation

All the software requirements assigned to ZVAD are satisfied by ZVAD_Task.c.
The software requirements can be traced on function level !.
For more details please see Requirements Implementation chapter.
ZVAD component doesn't have any non-functional requirements.

Software Unit Interfaces

We have no interfaces between units because there is just one unit in ZVAD.

Software Unit Interaction

Since ZVAD has one unit, there is no interaction between software units.

Dynamic Behavior

ZVAD has no dynamic behavior between SW units.

Tasks

No OSEK Tasks needs to be configured by the module.

Interrupts

ZVAD uses no interrupts.

Events

ZVAD uses no events.

Operating Modes

Not applicable because ZVAD component runs in high-power mode. At the moment, ZVAD doesn't require a special power mode.

Power Modes

ZVAD runs only in high power since it doesn't request special power modes.

Exception and Error Handling

Error status signals are provided by ZVAD to the the 3rd party libraries used to handle various error scenarios related to CAN messages.

Synchronization

Since the component is designed to run on a single core, the request variables are protected by means of critical sections Suspend/ResumeAllInterrupts.

Operations

Operation are available under the section: File section.

Macros

ZVAD macros are available in the Functions section: Macros section.

Algorithms

ZVAD component doesn't make use of or implements any particular algorithm.

Data Structures

No complex data structures used apart from those imported from other functionalities. In such cases, descriptions are given at the coresponding function that uses them.

Interoperability

Interoperability is set to low because ZVAD depends on other components.

Interaction

Interaction is set to low because the component has a reduced number of non RTE dependencies. See VW_ICAS1.Chorus.All_Cores.SoftwareDecomposition.ApplicationLayer.ZV_Lib.SW Architecture.ZVAD.ZVAD Static.ZVAD Dependecies.

Criticality

Criticality is set to low because ZVAD is a QM component, the level of interaction is low and component was not integrated recently, the criticality is Medium.

Technical complexity

Technical complexity is set to low because so far ZVAD doesn't implement any algorithm only integrates the library functions and interaction component is low.

Risks

Risks is set to low because ZVAD has only one SW unit. See VW_ICAS1.Chorus.All_Cores.SoftwareDecomposition.ApplicationLayer.ZV_Lib.SW Architecture.ZVAD.ZVAD Dynamic.ZVAD Init and Cyclic.

Testability

Testability is set to medium because ZVAD can be tested on SWATT / PLAST and basic tests should be performed on the target.
