Internal Component Design

Internal Component Design

The component has the following files that are developed by Continental:

  • DWAX_Task.c - This file will init, start run and stop the DWAX functionality
  • DWAX_Task.h - Header file for DWAX_Task.c
  • DWAX_NvRam.h - Header file is needed for connecting memory and lib
  • DWAX_ExtraDefinitions.h - This headerfile controls the overwriting of generated S/R autosar_functions
  • DWAX_DataTypes.h - This headerfile provides the datatypes used to provide the S/R Autosar functions for coding, fault memory, etc.
Internal Component Design

Requirements Allocation

All the software requirements assigned to DWAX are satisfied by DWAX_Task.c .

Software Unit Interfaces

We have no interfaces between units because there is just one unit in DWAX.

Software Unit Interaction

Since DWAXWrapper is made up of only one unit, there are no interfaces between software units to describe.

Interrupts

No OSEK event needs to be configured by the module.

Operating Modes

Not applicable because DWAX component runs in high-power mode. Currently, the component does not require a special power modes.

Power Modes

As mentioned in above chapter, the current functionality is available only in high-power mode.

Exception and Error Handling

Error status signals are provided by DWAX to the the 3rd party libraries used to handle various error scenarios related to CAN messages. The functions are handling errors via return value (e.g. RTE_E_OK). DWAX component reports internal errors on DTC level. For more details, see DWAX_Task.c

Synchronization

Since the component is designed to run on a single core, the concurrent access of the component is not required.

Operations

DWAX operations are available in the Functions section.

Macros

No complex macros or function like macros used.

Algorithms

No complex algorithms are used.

Data Structures

No complex data structures used. A structure that stores internal DWAX data is used. For more details, please see structure page.

Evaluate Software Detailed Design

Interoperability Interaction Criticality Technical complexity Risks Testability
Low Low Medium Medium Low High

Interoperability

Interoperability is set to low because DWAX depends on other components.

Interaction

Interaction is set to low because the component has a reduced number of non RTE dependencies.

Criticality

Since DWAX is non-safety, the interaction is Low and the component was not integrated recently, the criticality is medium.

Technical complexity

The technical complexity is medium because DWAx implements a specific behavior.

Risks

The component DWAx is composed of only one unit and no complex algorithms are implemented therefore the risk is low.

Testability

The component DWAx requires other tools (like PLAST) besides SWATT for 100% tests coverage but all tests can be automated. So, the testability level is high.
