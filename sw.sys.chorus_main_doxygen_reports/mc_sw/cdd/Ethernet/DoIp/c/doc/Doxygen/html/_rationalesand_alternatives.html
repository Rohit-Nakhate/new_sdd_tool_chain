Rationales, Alternatives and Constraints

Rationales
Chorus PU must address received diagnostics messages over ETH, which needs to be processed by itself or routed to at the ECU connected to a network in which the Chorus is also a node.


Alternatives
There were 2 possible alternatives regarding the routing algorithm:

  • Cut-Through - without intermediary buffers. 
  • Store and forward  - with intermediate storage.

  In theory the 1st option brings bigger routing speed, but it was not possible to be used as the diagnostic frames needs to be first stored and processed and afterwards routed to a specific node.
Therefore the second option was used (Store and Forward).


Constraints
According with the VW IP network requirements all the communication is done over IPv6, including DoIP messages.
DoIP from Chorus contains an exception to this requirement, raised from the limitation of Chorus GW Bootloader to handle only IPv4 format. As a consequence, DoIP Chorus requires a local IPv4 address and socket(s) to bind at, for DoIP communication with the bootloader.
