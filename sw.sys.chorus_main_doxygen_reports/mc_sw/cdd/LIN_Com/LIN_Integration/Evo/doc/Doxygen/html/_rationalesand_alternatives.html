Rationales and Alternatives
  • In the past there were functions that were duplicated for each LIN bus. It was decided to spend time in refactoring the code so that it is more generic so that LIN can be added and removed more easily.
  • Instead of having functions multiple functions LinInt_ScheduleTable_<verb><table-type><request> for each schedule type fewer functions could be created by having the table-type as a parameter.
    For example instead of LinInt_ScheduleTable_RequestAlternative, LinInt_ScheduleTable_RequestFlash there could be something like LinInt_ScheduleTable_Request(scheduleType type).
    However usually one type of LIN table is required by different components this way is better for the 'separation of concerns'. Event if we have multiple functions they are small so the memory footprint is neglijable.
