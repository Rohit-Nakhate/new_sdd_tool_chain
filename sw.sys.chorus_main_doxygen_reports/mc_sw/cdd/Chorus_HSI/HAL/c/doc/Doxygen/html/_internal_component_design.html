Internal Component Design

Component modules

The component is splitted into the follwing SW units:

  • hal.h - Export the exteral interfaces to be used by the other modules.
  • hal_cfg.h - General configuration for items from the HAL module.
  • hal_base.h - Provides the declaration of the types and functions used by other modules. Defines the structure for the generated code.
  • hal_base.c - Contains the implementation of the functions used as basis for other modules. The implementation is based on the active configuration.
  • hal_defines_gen.h -
  • hal_cfg_gen.h -
  • hal_cfg_gen.c -
  • hal_common.h - Header file for hal_common.c. Provides declaration of functions that need low level access common to all Chorus uC types.
  • hal_common.c - Source file for HAL common part. Contains implementation of functions that need low level access common to all Chorus uC types.
  • hal_variant.h - Provides declaration of functions that need low level access specific to one Chorus uC variant.
  • hal_variant.c - Contains implementation of functions that need low level access specific to one Chorus uC variant.
  • gw_dev_msg.h - Provides declarations of function for sending GW developer messages.
  • gw_dev_msg.c - Contains implementation of functions for sending GW developer messages.

Tasks

The module does not provide its own tasks. The provided functions are needed by other modules.

Interrupts

The module does not provide its own interrupts.

Events

The wakeup events detection is done in the module. See HAL_WakeupReason_CheckNewEventAndClear.

Operating Modes

There are no specific operating modes for this module.

Power Modes

The HAL component operates only in High Power Mode but prepares the configuration of certain ports for sleep mode.

Exception and Error Handling

There are no specific handling of errors such as DEM events. The functionality for this modules does not rely on external input.

Synchronization

No syncronization is necessary for the moment. Only one function should update the global variables most of them during startup.

Evaluate Software Detailed Design

Interoperability Interaction Criticality Technical complexity Risks Testability
Medium High Medium High High High

Interoperability

Interoperability is set to medium because a simple wrapper is required to exchange data.

Interaction

Interaction is set to high because the number of dependencies from dependency diagram is greater than 7.

Criticality

It has medium critcality as it can cause issues with the power consumption of the current but is not expected to have an influence on the driving.

Technical complexity

The technical complexity is high because HAL is split in SW units and has interactions with other components.

Risks

The number of SW Units of HAL is bigger than 5, therefore the risk is high.

Testability

The component HAL requires other tools (like PLAST) besides SWATT for 100% tests coverage but all tests can be automated. So, the testability level is high.
